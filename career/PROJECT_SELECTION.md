# Project Selection Framework

---

## Purpose

The purpose of project selection is to maximize long-term career leverage, skill development, reputation, and learning while minimizing wasted effort.

Time is the most limited resource. 

Not every opportunity deserves execution. Even excellent execution cannot fully recover from poor opportunity selection. This framework helps answer the critical question: **"Should I invest the next 50–500 hours into this?"**

---

## Core Philosophy

- **Prefer leverage** over activity.
- **Prefer depth** over superficial breadth.
- **Prefer real-world impact** over academic exercises.
- **Prefer projects that compound** over time.
- **Prefer opportunities that create future opportunities** (e.g., maintainership, referrals).
- **Prefer learning-rich environments** with strong feedback loops.
- **Prefer execution** over prestige alone.

---

## Selection Criteria

Evaluate every potential opportunity against the following five vectors:

### 1. Skill Growth
Will this project improve your:
- Engineering and coding ability?
- System design and architecture reasoning?
- Open source workflow and tooling mastery?
- Technical communication and collaboration?

### 2. Portfolio Value
Will this project produce visible evidence worth showing to:
- Recruiters seeking backend and systems talent?
- Open-source maintainers looking for serious contributors?
- Future co-founders or engineering collaborators?

### 3. Career Leverage
Will this project improve your:
- Professional network and developer relationships?
- Personal reputation and visibility in specific domains (e.g., CNCF, Kubernetes)?
- High-quality inbound opportunity flow?

### 4. Technical Depth
Does this project expose you to:
- Real-world production-grade systems?
- Non-trivial distributed systems or performance problems?
- Critical design decisions and trade-off analyses?
- Concurrency, databases (e.g., PostgreSQL), or cloud-native scale?

### 5. Compounding Potential
Will this project build reusable capital, such as:
- Foundation for future projects or technical articles?
- Credibility that makes future applications (e.g., GSoC, LFX) significantly easier?
- Long-term assets that generate interest and opportunities passively?

---

## Minimum Thresholds

An opportunity should generally satisfy at least one of:
- **Exceptional skill growth** (9-10 score)
- **Exceptional career leverage** (9-10 score)
- **Exceptional technical depth** (9-10 score)
- **Exceptional compounding value** (9-10 score)

*If an opportunity scores moderately across all dimensions but excels at none (e.g., scoring 7/10 across all five categories for a total of 35/50), reconsider whether it deserves a significant time investment. A project with exceptional spikes is generally a much higher-leverage opportunity than one that is merely average across the board.*

---

## Scoring Framework

Evaluate the opportunity by assigning a score from **1 to 10** for each of the five criteria:

| Criterion | Score | Rationale / Notes |
| :--- | :---: | :--- |
| **Skill Growth** | /10 | |
| **Portfolio Value** | /10 | |
| **Career Leverage** | /10 | |
| **Technical Depth** | /10 | |
| **Compounding Value** | /10 | |
| **TOTAL SCORE** | **/50** | |

### Score Interpretation
- **45–50 → Exceptional:** A rare, must-do opportunity. Drop other low-leverage activities to accommodate this.
- **40–44 → Strong:** Highly valuable. Strongly align with your roadmap and execute if resources permit.
- **35–39 → Good:** Solid project. Proceed if it directly fills a specific gaps in skill or visibility.
- **30–34 → Conditional:** Only pursue if there are no higher-scoring options and you have surplus bandwidth.
- **<30 → Reject:** Do not commit. The opportunity cost is too high.

## Opportunity Categories

### Open Source
*High value if:*
- The project has active, responsive maintainers and a strong review culture.
- The project serves real users with active deployments.
- It aligns with long-term goals (e.g., CNCF projects like Harbor, Kubernetes).
- The codebase provides deep learning opportunities.

### Side Projects
*High value if:*
- They solve real, practical problems rather than being toy duplicates.
- They require non-trivial engineering depth (e.g., database internals, compilers, network tools).
- They produce reusable assets or packages.
- They showcase clean code design, tests, and documentation.

### Certifications
*High value ONLY if:*
- They are recognized and respected by industry hiring managers (e.g., CKA, CKAD).
- They demonstrate a meaningful, verified baseline skill.
- They actively fill a specific credibility gap on your resume.
- *Avoid certificate collection for the sake of credentials.*

### Fellowships & Mentorship Programs
*High value if:*
- Mentors are active and respected in the industry/community.
- The program leads to meaningful, production-grade project work.
- The community remains active and useful after the program's completion.
- The work produces public artifacts and open-source contributions.
- The experience creates future high-leverage opportunities (e.g., referrals, jobs).
*Examples:* LFX Mentorship, GSoC, MLH Fellowship, Summer of Bitcoin.

### Competitions
*High value if:*
- They challenge you to build real systems under constraints.
- They provide avenues to build a high-caliber network.
- They produce durable, high-quality portfolio artifacts.

---

## Red Flags

**Reject opportunities immediately if they:**
- Exist purely for resume padding with no real substance.
- Offer zero technical depth or learning potential.
- Rely on inactive, hostile, or unresponsive communities.
- Require massive time and effort for minimal return.
- Duplicate experience you have already mastered.
- Distract from existing, higher-leverage commitments.

---

## High-Leverage Indicators

**Look for strong signals that indicate high return on investment:**
- **Real Users:** The project solves active problems for real people.
- **Real Maintainers:** Experienced devs who offer high-quality, constructive code reviews.
- **Real Production Systems:** Exposure to real-world operational constraints and scale.
- **Public Visibility:** Open source commits or public presentations that speak for themselves.
- **Strong Engineering Culture:** Excellent testing standards, clean documentation, and automated CI/CD pipelines.

---

## Decision Process

Before committing your time, follow this systematic evaluation process:

1. **Score the Opportunity:** Run it through the scoring matrix objectively.
2. **Conduct Opportunity Cost Analysis:** Explicitly outline what you are giving up.
3. **Compare Alternatives:** Stack it against other projects or options currently available.
4. **Sleep on the Decision:** Give your mind 24–48 hours to process the choice.
5. **Re-evaluate:** Score it again with fresh eyes.
6. **Commit fully:** If selected, allocate the necessary hours and focus entirely on execution.

---

## Opportunity Cost Analysis

Time spent on one project is time taken away from another. Ask yourself:

1. *What will I NOT be able to do or learn if I choose to invest my next 100 hours here?*
2. *Would I still choose this project if a highly desirable LFX mentorship or core CNCF opportunity appeared tomorrow?*
3. *Is this the single highest-leverage use of my time right now?*

---

## Review and Reflection

After completing a project or walking away from an opportunity, reflect on the decision:

- **What worked?** Did the project deliver the expected skill growth or leverage?
- **What didn't?** Were there unexpected roadblocks or low-return efforts?
- **Was the selection correct?** Did the scoring accurately predict the project's value?
- **Would I choose it again?** Knowing what you know now, would you still make the same choice?
- **What signals did I miss?** Identify missed red flags or leverage indicators to refine future selections.

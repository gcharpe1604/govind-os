# Research and Repository Discovery

## Trigger

The user asked to research a technical question, orient to an unfamiliar repository, find an opportunity, or compare approaches.

## Inspect

1. Define the decision the research must support and the freshness required.
2. Start with repository-local instructions, code, tests, manifests, and authoritative project docs.
3. Search before opening; trace entry points and key symbols instead of scanning entire trees.
4. Use current primary sources for unstable external facts, versions, APIs, security, or policy.

## Decide

- Separate primary evidence, secondary context, inference, and unknowns.
- Prefer sources closest to the actual behavior.
- Stop adding sources when new evidence no longer changes the conclusion.
- For opportunity discovery, require a reproducible problem, affected users or maintainers, scope, and a plausible acceptance path.

## Act

Research and report only unless the user requested implementation or external action. Do not create a framework when a concise conclusion or existing project process is enough.

## Verify

Cross-check consequential claims against code, runtime evidence, or current primary documentation. Confirm dates and versions. Test proposed commands or APIs when safe and relevant.

## Stop

Return the answer, strongest supporting evidence, uncertainty, and next action. If blocked by inaccessible evidence, state what could not be verified and avoid presenting stale memory as current fact.
